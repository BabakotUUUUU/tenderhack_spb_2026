## Быстрый старт (Docker)

```bash
# 1. Клонировать/распаковать проект
cd tenderhack_spb_2026

# 2. Дать права на запись для SearXNG (Linux/Mac)
chmod 777 searxng/

# 2a. На Windows — просто пропустить шаг выше

# 3. Собрать образы и запустить всё одной командой
docker compose up --build

# Или через Makefile (если make доступен):
make fresh
```

Порядок старта (автоматический, благодаря healthchecks):
```
SearXNG  →  Backend  →  Frontend
```

SearXNG ждёт ~20 сек до готовности, Backend ещё ~15 сек.
Итого первый запуск: **~3–4 минуты** (скачивание образов + сборка).

### Адреса после запуска

| Сервис | URL |
|--------|-----|
| **Веб-интерфейс** | http://localhost:3000 |
| **API (Swagger UI)** | http://localhost:8000/docs |
| **API (ReDoc)** | http://localhost:8000/redoc |
| **SearXNG** | http://localhost:8080 |

### Полезные команды

```bash
make logs          # логи всех сервисов
make logs-backend  # только backend
make status        # статус контейнеров
make down          # остановить всё
make health        # проверить health-endpoint
```

## Локальная разработка (без Docker)

### Backend

```bash
# 1. Создать виртуальное окружение
python -m venv .venv
source .venv/bin/activate       # Linux/Mac
.venv\Scripts\activate          # Windows

# 2. Установить зависимости
pip install -r backend/requirements.txt

# 3. Запустить SearXNG отдельно (нужен Docker для 4-го источника)
docker compose up searxng -d

# 4. Запустить backend
cd backend
SEARXNG_URL=http://localhost:8080 uvicorn app.main:app --reload --port 8000
```

### Frontend

```bash
cd frontend
npm install
npm run dev
# Откроется на http://localhost:5173
# Запросы к /api/* проксируются на http://localhost:8000 (vite.config.js)
```

### Почему SearXNG для 4-го источника?
- **Self-hosted** — работает на наших серверах, без внешних API-ключей
- **Не фиксированный** — агрегирует DuckDuckGo + Bing + Google динамически
- **Open-source** (AGPL-3.0) — соответствует условиям хакатона
- **JSON API** — удобная интеграция с backend

## NLP и ML

### Что используется (без внешних API, только open-source)

#### 1. `pymorphy3` — морфологический анализатор русского языка
- **Тип**: rule-based (словарный), не нейросеть
- **Размер**: ~15 MB
- **Скорость**: < 0.5 мс на слово
- **Задача**: Лемматизация — приведение слова к начальной форме
  ```
  «ноутбуки» → «ноутбук»
  «резины»   → «резина» → синоним → «шины»
  «принтеров» → «принтер»
  ```
- **Обоснование**: единственный зрелый морфологический анализатор для русского языка, работающий полностью офлайн и без ML

#### 2. `rapidfuzz` — нечёткое сравнение строк
- **Тип**: алгоритмический (расстояние Левенштейна), не нейросеть
- **Размер**: ~1 MB
- **Скорость**: < 1 мс на слово
- **Задача**: Исправление опечаток по схожести строк
  ```
  «ноутбукк» → 95% схожесть → «ноутбук» ✓
  «кросовки» → 92% схожесть → «кроссовки» ✓
  ```
- **Порог**: 82% — эмпирически подобранное значение, исключает ложные срабатывания

#### 3. Словарь синонимов (встроенный)
- **Тип**: статический справочник, нет ML
- **Размер**: < 1 KB в памяти
- **Задача**: Расширение запроса синонимами категорий хакатона
  ```
  «резина» → «шины», «покрышки», «tires»
  «лаптоп» → «ноутбук», «laptop»
  «мфу»    → «принтер», «сканер», «копир»
  ```

### Почему НЕ используем тяжёлые нейросети?

Задание явно требует: *«Предпочтение будет отдаваться легковесным решениям с высокой скоростью инференса при ограниченных вычислительных мощностях»*.

Альтернативы, которые **намеренно не применяются**:

| Модель | Размер | Зачем бы | Почему не берём |
|--------|--------|----------|-----------------|
| ruBERT-tiny2 | ~120 MB | семантический поиск | избыточно для словарного поиска |
| Word2Vec (ruscorpora) | ~600 MB | синонимы через эмбеддинги | словарь даёт тот же результат быстрее |
| LLaMA / Qwen-local | 4–70 GB | понимание запроса | запрещено (тяжёлые модели), нет GPU |

### Пайплайн обработки запроса

```
Пользователь: «ноутбукк HP 15»
      ↓
[pymorphy3] Лемматизация каждого слова
      ↓
[TYPO_MAP] Прямой поиск в словаре опечаток
      ↓
[rapidfuzz] Нечёткий поиск по известным словам (порог 82%)
      ↓
Исправлено: «ноутбук HP 15»
      ↓
[SYNONYM_MAP] Поиск синонимов
      ↓
Варианты: «ноутбук HP 15», «laptop HP 15»
      ↓
Парсеры WB + Ozon + YM + Runet (параллельно)
```

### Что можно улучшить (если будет время/ресурсы)

Для более сложных задач (понимание намерений, извлечение характеристик из запроса) можно добавить:
- `natasha` + `slovnet` — NER для извлечения бренда/модели из запроса (~40 MB)
- `sentence-transformers` c моделью `paraphrase-multilingual-MiniLM-L12-v2` (~120 MB) — семантическое сравнение запроса с найденными товарами

## Rate Limiting

### Обоснование выбранных задержек

| Источник | Задержка | Причина |
|----------|----------|---------|
| Wildberries | нет | официальный публичный API, без rate limiting |
| Ozon | 1.5–5 сек | защита от bot-detection (429 при < 1 рек/сек) |
| Яндекс Маркет | 2–4.5 сек | агрессивная защита, Captcha при быстрых запросах |
| Руnet/SearXNG | 1.5–3.5 сек | задержки к конечным сайтам |

**Дополнительные механизмы:**
- Ротация User-Agent из 5 вариантов реальных браузеров
- Экспоненциальный backoff при ошибке 429 (3×3 = 9 сек, потом 9×3 = 27 сек)
- `asyncio.Semaphore(3)` — не более 3 одновременных запросов к одному домену
- Реалистичные заголовки браузера (Accept, Accept-Language, Sec-Fetch-*)

**Итоговое время поиска**: 5–15 секунд в зависимости от задержек парсеров.

## Известные ограничения

1. **Яндекс Маркет** и **Ozon** активно используют антибот-защиту. При первом запросе данные могут отсутствовать или быть неполными. WB и Руnet работают стабильно.

2. **Изображения WB** используют формат WebP и могут не отображаться в очень старых браузерах.

3. **Первый запрос** занимает 5–15 сек из-за задержек rate limiting. Повторные запросы той же категории могут быть быстрее (HTTP keep-alive).

4. **SearXNG** на первом запросе после старта контейнера может отвечать 5–10 сек — это нормально (cold start).

5. Решение **не требует авторизации** — соответствует условиям хакатона.

## BPMN-схема

```
Старт: Пользователь вводит запрос
  ↓
[NLP Gateway] → Лемматизация (pymorphy3) → Исправление опечаток (rapidfuzz)
  ↓                                              ↓
Запрос исправлен?                         Отображение баннера коррекции
  ↓
[Параллельный шлюз] ─────────────────────────────────────────────
  │          │               │                    │
[WB API] [Ozon HTML]  [YM HTML]          [SearXNG → Runet]
  │          │               │                    │
  └──────────┴───────────────┴────────────────────┘
                [Joining Gateway]
                      ↓
              Агрегация результатов
                      ↓
              Группировка по источникам
                      ↓
              Ответ клиенту (JSON)
                      ↓
              Отображение карточек товаров
                      ↓
                    Конец
```

**Используемое ПО:**
- Python 3.12, FastAPI, uvicorn — backend
- React 18, Vite, CSS Modules — frontend
- Nginx — статика и proxy
- SearXNG — self-hosted поисковый агрегатор
- Docker Compose — оркестрация
- pymorphy3, rapidfuzz — NLP
- httpx, beautifulsoup4, lxml — парсинг
